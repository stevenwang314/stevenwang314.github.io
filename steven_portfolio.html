Song recommendations using Spotify API

A person has songs that they like to listen over and over. But what if they want to listen to new songs?

Assuming the user has a Spotify Account, the goal of this app is for the user to be able to compile a list of songs they like. The user can either use their own Spotify Account to use their playlist as their list or they can use a search engine in Spotify to grab songs of their choice. In the contrary, the user can create songs they dislike of their choice.

Once the user decides what the like or dislike, they can pick up to 5 songs or artists as a selector to feed to Spotify to generate a list of songs that will be recommended from there. In addition, having more songs on the list does affect the song recommendations even though they are not selected. The user can filter out songs based on popularity. Songs that are not popular will never be selected.

The user can repeat this step over and over by changing the 5 songs or artists of their own liking. The list of recommended songs will only add songs that are not on the list.

When the user decides he/she is ready to accept the list, they can click a button that adds ALL of the recommended songs except the songs they dislike into a new playlist directly at an instant. The user then can play the playlist without having to add the songs manually which takes a excruciating amount of time.

Implementations

Language: Javascript, HTML, and CSS Libraries: React, Bootstrap, Jquery, Spotify API, RESTful API, and Browser’s local storage

In order to use the Spotify API and their RESTful library, an access token is required. To get an access token, the program must follow the Authorization Guide by first obtaining an access token through (in this project) the Authorization Code. Second, I must create a Spotify app on their dashboard to obtain several sensitive information such as Client Key and Client Secret and finally, I need to use an existing Spotify Account to gain authentication to finally get the Access Token.

As the Client ID and Client Secret cannot be shown in source code, I need a way to hide it from sight. To securely hide the sensitive Client ID and Client Secret, it is stored in a hidden environment variable file behind the server-side part of the app through Node.js. It uses Express to accept request from our own client-side where it calls GET/POST methods to our server side. There the program has two API functions which we implemented that can authorize the users Spotify account into the app and to obtain an access token by using Axios library. Both utilizes the Client ID and Client Secret needed to authenticate the user and then get an authentication code which is needed to hook into Spotify API. Once we get the access token, we finally can use it to call our API calls from Spotify.

In this program, getting the access token is automatically called once the user is able to click the refresh permission button but also an additional time once the access token expires, which is in 1 hour, as there is also a refresh token to get another access token which only happens once per authentication.

In the front-end part of our app, we use the fetch() function to call our API request with our access token to Spotify API to be able to search for songs, artists, albums, recommending songs, getting song audio features, and then adding songs to playlist endpoints. To be able to constantly update the dynamic list on the webpage whenever the lists are changed from user, it uses React whenever we modify the list to update the displayed dynamic list.

The input we feed it to the Spotify Song recommendation algorithm uses the average of all songs features that are on the favorite songs list from the user followed by the 5 “seeds” from the users’ songs/artists to the generate recommended songs endpoint. Finally, once the song recommendation list is made, we use the save song to playlist endpoints to directly add them to the playlist directly.

Tower Defense

A tower defense game with 20 towers each having specific unique properties to handle against 18 different enemies in a map. There are enemies that can weaken towers by making them shoot slower, weaker projectiles, reduce their enemy detection range, as well as not letting them do anything at all. There are also enemies that help boost enemies speed and even force towers to attack a specific enemy. For towers, there are some towers that requires manual casting to deal powerful attack as well as some towers that boost other towers performance.

Other than that, normal tower defense rules still apply. The player spends gold to build towers as well as upgrading towers to improve their stats. The goal is to prevent the enemies reach the exit path. Each enemy that escapes to the end of the path costs 1 health and the game ends when all health is lost. With each new wave features newer enemies as well have an ever increasing health. There isn’t an end to the game but to get as far as possible.

Implementations

Language: C++, Libraries: C++ Standard Library, Windows32 , Direct2D, XAudio2, MSXML APIs

The libraries used to build this project were Microsoft’s Windows32 API to create a window program, Direct2D API to render images, shapes, DirectWrite API to render Text, and XAudio2 to play audio.

Each Towers, Enemies, and Projectiles uses Object Oriented Programming (OOP) technique and are assigned in classes. To differentiate different towers and enemies, each tower and enemy uses the inheritance technique that inherits the base turret/enemy class and override some of its properties for its own custom implementation.

Angled collision is common especially when turrets are facing at an angle. For the projectiles to be able to handle angled collision correctly, it uses the Separate Axis Theorem which allows concave shapes collisions by using each of its vertices on two objects to project an axis between each other.

Another issue is where when there are more towers and enemies on the screen as the game progress which results in more redundant Collison checks which leads to reduced performance of the program. To reduce the collision checks, I have implemented a Spatial Hash which is a hash table that divides parts of the world space into a list for that particular chunk. To be able to get objects at that particular rectangle, we access the Spatial Hash using positions from 4 corners and aggregate it into an object list. Instead of checking all enemies (regardless of position), we only check only enemies that are roughly near a particular rectangle which improves our game performance.

Sudoku

A number placement puzzle game where the goal is to fill a partially completed grid using numbers from 1 to 9 to finish the puzzle. The sole rule of this puzzle game is that each row, column, and region in a 9x9 grid is to contain all the numbers from 1 to 9.

The person can use numbers 1 to 9 to fill in the empty cells in hopes of filling the entire puzzle. Sometimes it can’t be done easily, in which the player can mark candidates from 1 to 9 in which tells the player that they can be possible numbers that fill in the cell. All advanced techniques utilized candidates which narrows down possible candidates to a few in which the user can finish the puzzle off easier. Or if the user wants to save time, the person can use auto-fill candidates to fill ALL possible candidates in the entire puzzle so they don’t have to write themselves.

In case of an error, the program will detect errors whenever the player inputs an incorrect number such as two numbers in the same row, column, or region. An undo/reverse button to either redo or revert changes made into the puzzle.

There are many sample puzzles that can be loaded from an xml file for the player to solve. If that is not enough, the system can generate a new puzzle with a unique solution of its own providing nearly infinite amount of sudoku puzzles a person can solve.

Implementations

Language: C++, Libraries: C++ Standard Library, Windows32 , Graphics Development Interface (GDI), MSXML APIs

The libraries used to build this project were Microsoft’s Windows32 API to create a window program and draw the user interface (UI), Graphics Development Interface (GDI) to render images, shapes, and text.

In order to solve any sudoku puzzles (including the hardest sudoku puzzles out there), I have used the modified Backtrack Algorithm. The modifications to the Backtrack Algorithm are that first it will eliminate candidate numbers that does not fit in either row, column, or region in that particular cell. Next it will identify Naked/Hidden Singles and Intersection Row/Column techniques that will further reduce candidates possible. This significantly improves the algorithm speed as there will be less possible cases possible in a particular possible.

To generate a brand new sudoku puzzle, first the algorithm will remove all numbers from the sudoku puzzle and then shuffles the order of candidates in all 81 cells. Then, from top-left cell to the bottom-right cell. The algorithm will choose the first cell and fills it out normally while eliminating that particular number in its row, column, and region cell. This continues until the puzzle is finished or until there is no possible candidate cells at that location. If there aren’t any possible candidate cells, the algorithm will backtrack, going back to the previous cell and remove the select candidate cell from earlier and pick the next candidate cell in the list. Once the puzzle is fully filled, it will remove numbers at random cell location and attempts to solve the puzzle using the modified backtrack algorithm to solve it back but this time, the modified backtrack algorithm continues after getting a solution so that it can find more than 1 solution in that puzzle in which it will backtrack normally and continue on. If the removed number causes the puzzle to get 2 solutions or more, the number erased will be put back and will no longer be selected for removal anymore and the removal continues. This removal continues until all cells are considered or until it removes an specific amount of cells.

To be able to solve sudoku puzzle using numbers logic, it is much harder to implement then modified Backtrack algorithm. To be able to get started, I had to create several arrays which are number of candidates, list of singles, pairs, triples, and quads, as well as candidates that remain in all row, column, and region. This allowed me to make Naked/hidden single, pairs, triples, and quads, intersections (pointing/claiming), X, XY, XYZ, W wings, Swordfish, and Simple Coloring. This has been extremely challenging for me as it is extremely hard to debug and find errors due to a 2d array of 9x9 making it hard to find which index were changed and so on.

Seizure Flash Blocker

Made in Hackathon of Spring 2019 in 48 hours, the goal of the project is to replace quick flashing strobe of lights in certain videos into a static color to protect the viewer from collapsing from a seizure. In this program we used Google Chrome and a Youtube video (as demonstration) from a clip from Pokemon (Dennō Senshi Porygon) that caused hundreds of people to be sent to hospitals due to seizures caused in that episode.

A seizure (or a photosensitive epilepsy) is made from constantly flashing colors or regularly repeating patterns

Implementations

Language: C/C++, Libraries: C++ Standard Library, Windows32 , Graphics Development Interface (GDI)

The program is run in the background, there is no window that shows up that shows up

It uses an algorithm which determines the aggregate color difference which compares the difference of color between the current frame and the last frame. The color has three channels of red, green, and blue dimension which is then pointed in space. We can compute the distance between two colors through the distance formula. If the distance is bigger, it is shown that the color is changed extremely. This is for one pixel. To be able to cover the entire screen, we have to sum the color distances over the entire region to compute an aggregate between two frames.

It records number of times within 200ms period that color difference reaches a specific threshold. If this threshold activates 3 times, it is marked as flashing. We used GDI (Graphics Development Interface) to draw a red rectangle over the region.

System on Chip Design - Tramel Blaze

Tramel Blaze is a modified version of PicoBlaze architecture where it is able to execute 16bits of instructions compared to PicoBlazes 8 bit instructions. It contains components of 16 bit registers, SRAM (Static Random Access Memory) with 32,768 address, TSI (Technology Specific Information), and UART (Universal Asynchronous Receiver Transmitter) modules that when used, it can store/load data from memory as well as receiving/transferring data from another device. Our emulated design is then programmed onto a Nexys4 Board. There, it is fed lines of assembly language instructions and its output appears on the 8 digit LED. The buttons go through each register and also traversing through instructions

Implementations

Language: Verilog / Assembly Language

AISO – The Asynchronous Input Synchronous Output is a module that turns an Asynchronous signal into a Synchronous Output. The main use of the module is to prevent an unstable signal know as metastability.

TSI – The Technology Specific Instantiation is a module that makes sure that the I/O of the chip meets electrical and timing requirements of the external interface.

PED – The positive edge detect simply detects whether a bit changes from low (0) to high(1)

Address Decoder – The address decoder outputs control units based on the given input varying from address and read/write strobe.

UART - The Universal Asynchronous Receive Transmitter has two component: The transmit side and the receive side.

The transmit engine uses a Baud Decoder to translate 4 bits of input to a given selected Baud Rate. The larger the baud rate, the less clock cycles it takes to reach the Bit time counter. See the table in Chip Specification for more details.

It uses the Shift Register to shift 1 bit to the right. The Least Significant bit is then sent to the TX signal. The contents contain a start bit and stop bit which is a fixed set of bits that tells the UART to start/stop processing the bits.

The receive engine is much more complex than transmit engine. It is not a voluntary process and will wait until the transmit data from the other peripheral starts sending data to the receive side of this peripheral.

Not all data will be transferred successfully from one side to another. We have to flag Parity, Framing, and Overflow errors as a way to check whether we are able to successfully transfer all of its bits from the other peripheral transmitter data to our receiver data.

Single Cycle Mips Processor

A Single Cycle Mips Processor is used to execute assembly instructions in one single clock cycle. There are many different instructions that affect registers and values in the memory address in the processor. We extended our Single Cycle Mips Processor by adding more instructions into the processor such as Load Word Increment (LWINC), Exchange (Xch), Multiple (MULT), Divide(DIV), Jump Bit (JB), Jump Carry( JC), and etc. By adding more instructions, we have to modify the structure of our processor in order to incorporate the new instructions to ensure that these extension instructions can be resolved with a single clock cycle. There is a disadvantage of adding more instructions in hardware, it cost more money to be able to contain additional hardware components but also it costs more power to be able to operate additional components. The choice is made by the person.

Implementations

Language: Verilog

Extended Instructions that are implemented to our MIPS processor.

Multiply (mult) multiplies a register by another register. To implement this we have to follow this algorithm to incorporate in our verilog module.

Divide (div) divides a register by another register. To implement this we have to follow this algorithm to incorporate in our verilog module.

Jump Bit (jb)- Jump to a specific address when a particular bit is set (1).

Jump Carry (jc)- Jump to a specific address when the carry flag is set (1).

Load Byte Unsigned (lbu)- Loads an immediate value into the register and then transform the value into unsigned modifying the most significant bit to always be 0.

Store Word Increment (swi)- Stores a immediate value from a register into memory, then the current address is incremented by 1.

Store Word Decrement (swd)- Stores a immediate value from a register into memory, then the current address is decremented by 1.

Load Word Increment (lwi)- Loads an immediate value into a register, then the current address is incremented by 1.

Load Word Decrement (lwd)- Loads an immediate value into the register, then the current address is decremented by 1.
